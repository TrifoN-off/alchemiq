# SQLite (dev and test tier)

alchemiq officially supports SQLite for **development, tests, and embedded
scenarios**.  PostgreSQL remains the production target: the promise of this
tier is that the test suite of your alchemiq/PostgreSQL application runs on
SQLite without Docker - with the exceptions listed in the feature matrix below.

Requires SQLite 3.24 or newer (2018; needed for `INSERT ... ON CONFLICT`).

## Quickstart

Install the driver and point `configure()` at a SQLite DSN - nothing else
changes:

```bash
pip install alchemiq[sqlite]
```

```python
import alchemiq

alchemiq.configure("sqlite:///./dev.db")        # file-based
alchemiq.configure("sqlite+aiosqlite:///:memory:")  # in-memory (tests)

await alchemiq.create_all()
```

Driverless `sqlite://` DSNs are normalized to `sqlite+aiosqlite://`
automatically (the same courtesy maps `postgresql://` to
`postgresql+asyncpg://`).  Foreign-key enforcement (`PRAGMA foreign_keys=ON`)
is enabled on every connection.  In-memory databases use a single shared
connection, so every session sees the same data - ideal for test fixtures:

```python
import pytest_asyncio

import alchemiq


@pytest_asyncio.fixture
async def db():
    alchemiq.configure("sqlite+aiosqlite:///:memory:")
    await alchemiq.create_all()
    try:
        yield
    finally:
        await alchemiq.dispose()
```

## What works

Everything not listed in the matrix below, including: CRUD and queries,
offset and cursor pagination, `bulk_upsert` (`INSERT ... ON CONFLICT` via the
SQLite dialect), relationships (`ForeignKey`, `OneToOne`, many-to-many),
soft-delete, optimistic locking, signals, the transactional outbox, caching,
health checks, and FastAPI integration.

Column types map automatically: `JSON` fields are stored as SQLite `JSON`
(not `JSONB`), `UUID4`/`UUID7` as `CHAR(32)` strings (your Python code still
sees `uuid.UUID`), integer primary keys are stored as `INTEGER` on SQLite
(`BIGINT` on PostgreSQL) so autoincrement works, and timezone-aware datetime
fields are stored as naive UTC and returned timezone-aware, matching
PostgreSQL behaviour.

## Feature matrix: PostgreSQL-only

| Feature | On SQLite |
|---|---|
| `Array` fields | `ConfigError` at table-creation time |
| `QuerySet.explain()` / `Repository.explain()` | `QueryError` |
| JSONB operators in filters | not available (storage/equality work) |
| Multi-worker relay | single worker only - see below |
| `DISTINCT ON` class features | not planned for SQLite |

Numeric/decimal columns round-trip through floats on SQLite (a SQLite
limitation) - exact-precision assertions belong in PostgreSQL tests.

## Relay: exactly one worker

SQLite has no row locking; the relay's `FOR UPDATE SKIP LOCKED` claim is a
no-op there.  A single relay worker behaves exactly as on PostgreSQL.  Running
two or more relay workers against one SQLite database can deliver the same
event more than once - do not do it.  (Delivery is at-least-once everywhere;
consumers must deduplicate on the message id regardless.)

## Migrations

Point the migration runner at SQLite with the `dsn` key (instead of the
host/database/username/password quartet):

```toml
[tool.alchemiq]
models = ["app.models"]

[tool.alchemiq.postgres]
dsn = "sqlite+aiosqlite:///./dev.db"
```

`alchemiq makemigrations` / `migrate` / `rollback` work as usual; on SQLite,
autogenerated `ALTER` operations are emitted as Alembic `batch_alter_table`
blocks (table-rebuild), which is transparent in practice.

Two caveats:

- Migration files autogenerated against SQLite may render dialect-specific
  column types for PG-flavoured fields.  Treat PostgreSQL as the source of
  truth for committed migration files; for pure-SQLite test setups prefer
  `alchemiq.create_all()`.
- The section name stays `[tool.alchemiq.postgres]` - it configures the
  relational (SQLAlchemy) backend, and the `dsn` key accepts any async DSN.
